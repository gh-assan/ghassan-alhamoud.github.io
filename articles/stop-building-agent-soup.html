Stop Building Agent Soup

Everyone's throwing agents at problems. Most of them are making things worse.

I've audited a dozen agent systems this year. Almost all of them suffer from the same problem: they're agent soup. A bunch of agents thrown into a pot with no clear structure, no defined communication protocols, and no governance. The result is a system that's harder to debug, harder to maintain, and less reliable than the monolithic code it replaced.

The Symptoms of Agent Soup

  • Hallucinatory handoffs — Agent A thinks it delegated to Agent B, but Agent B never got the message
  • Runaway loops — Two agents passing the same task back and forth until they hit the token limit
  • Competing objectives — Agent A is optimizing for speed while Agent B is optimizing for accuracy, and they're working on the same problem
  • Shadow agents — Nobody remembers what some agents do, but nobody wants to remove them either

If this sounds familiar, you have agent soup.

The Cure: Structured Agent Architecture

Here's what works instead:

1. Define Clear Boundaries

Every agent should have exactly one job. Not "help with customer support" but "classify incoming tickets by urgency and route to the appropriate queue." Single responsibility applies to agents too.

2. Explicit Communication Protocols

Agents should never "just talk" to each other. Define interfaces: input schema, output schema, error handling, retry policies. If you can't write a unit test for the handoff, the handoff is too vague.

3. An Orchestrator, Not a Democracy

Agents should not decide for themselves what to do next. An orchestrator — which can be a simple state machine or a directed acyclic graph — controls the flow. Agents are workers, not decision-makers.

4. Observability by Default

Every agent interaction should be logged with trace IDs. You should be able to replay any sequence of agent interactions to understand what happened. If you can't debug it in production, you shouldn't deploy it.

The Bottom Line

Agent architectures are powerful, but they're not magic. The teams that succeed with agents are the ones that treat them like distributed systems — because that's exactly what they are. Design them with the same rigor you'd apply to a microservice architecture, and they'll work. Throw them in a pot and hope for the best, and you'll get soup.
