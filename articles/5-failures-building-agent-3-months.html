I Spent 3 Months Building My Agent. Here Are the 5 Failures That Almost Killed It.

AI Agents Infrastructure Failure Architecture

I went all-in on building a personal AI agent.

Not a chatbot. Not a wrapper. An autonomous orchestrator that manages my inbox, scans my deals, writes my articles, and coordinates my pipeline.

I spent 3 months. I went through 5 different architectures. I killed 2 email accounts, wrecked a good machine, and found out that asking the assistant to set up itself is a loop so absurd it still makes me laugh.

Here's exactly what failed — starting with the one that hurt most.

1. The Email Layer: What Should Be the Easiest Part

I started with Gmail. Google's most mature product, right? If you can't get email right, you can't get anything right.

Three weeks in: Google deleted my account.

Not a warning. Not a rate limit. Full account termination. Because an autonomous agent reading and sending emails triggers every anti-abuse flag in their detection system. No appeal. No human to talk to. 50+ connected services, all dead overnight.

Fine. I switched to Hotmail. Same game, different enemy. Microsoft's authentication flow demands manual re-authentication every 6 hours. An agent can't pass a CAPTCHA. It can't click "Is this you?" in an authenticator app. Three days in, my agent was locked out while I was asleep.

The rescue: GMX — a German email provider nobody talks about. Combined with Thunderbird MCP. It works. Quietly. Without being detected as "suspicious." The agent never talks to SMTP directly — it talks to Thunderbird, which talks to the provider. A proxy layer that costs nothing and makes everything work.

The principle: The infrastructure that works for humans doesn't work for agents. Your agent needs boring providers — the kind of reliable-but-unsexy services that don't have aggressive bot detection. Security is a feature for humans. For agents, it's a blocker.

2. The Diagram That Never Worked

I had this vision: the agent reads architecture documents, generates beautiful Mermaid diagrams automatically. Beautiful, professional technical diagrams. Perfect for documentation.

Three approaches. Each one failed differently.

The first produced diagrams that looked like a 5-year-old drew them — wrong layouts, unreadable labels. The second generated valid Mermaid syntax but diagrams so busy they were useless. The third was decent — once — and never repeated it.

Why: Current models can't see what they make. No prompt will teach an LLM that lines overlap — because it's guessing at coordinates, not looking at the page. The feedback loop is broken at the architectural level. There's no "that box is too big" or "these lines cross unnecessarily" critique because the model doesn't experience the output as a human would.

The fix: Two-stage pipeline. Stage 1 generates the syntax. Stage 2 renders it and feeds it to a vision-capable model for critique. The critique feeds back into Stage 1. Repeat until it passes. Three iterations minimum. It works — but it costs more tokens than the article itself.

3. The Meta-Failure: Asking the Assistant to Set Up Itself

This is the one that still makes me laugh.

I was building an autonomous orchestrator as my personal assistant. And I thought: what if I make it capable of remotely configuring and deploying itself onto my machines?

The logic was sound. The execution was a 4-day engineering nightmare.

The sequence went: SSH into the remote machine → authenticate → discover what's installed → install dependencies → configure services → test.

Every step broke differently. SSH key management across machines. OS-level permissions — the agent couldn't install Docker because it didn't have root. It couldn't get root because it was running as a limited user. It couldn't change the user because it didn't have sudo. Each fix unblocked the next blocker — which was also broken. A directed acyclic graph of dependencies where every edge was a failure point.

I got as far as a partially working SSH-injection flow. Broke on the first real test: a missing Python dependency that wouldn't exist in any environment the agent wasn't designed for.

The real lesson: Not everything should be automated — at least not yet. The 80/20 rule cuts the other way here: the most "meta" automation problems — setting up the tool that sets up the tools — take 80% of the effort for 20% of the value. Deploying a complex agent system remotely is in the "hard" quadrant. Know when to just do it yourself.

4. The VSCode Dream That Collapsed

Another ambitious idea: let the agent control my development environment. Not by giving instructions — by controlling VSCode directly. Managing Copilot. Directing Cursor. Pushing commits. All from the agent layer.

This is the most-hyped combination in developer tools right now. And it doesn't work — not at my scale, not at this point in the industry.

The problem: two reasoning systems operating on the same surface area. The agent opens a file. VSCode auto-formats it. The agent adds a comment. Copilot suggests a completion that overwrites it. The agent tries to refactor. The IDE auto-completes something else. A coordination cascade that neither system can resolve — because neither knows the other exists.

I eventually fixed this by wrapping every IDE operation in a mutex: one actor at a time. The agent doesn't "control VSCode." It submits file patches, runs commands through a managed terminal session, and receives diffs through a structured output channel. Serialized. Explicit. No surprises.

The principle: The IDE must be a tool of the agent, not a peer. Until VSCode exposes a proper agent API with a conflict-resolution protocol, any deeper integration is premature for production.

5. The Model That Almost Wrecked Everything

I started with Gemini Flash as my primary model. Fast, cheap, available. What could go wrong?

On day 2, I gave it a simple instruction: "Read this email and summarize it. Never generate code."

It generated a 200-line Python script and tried to execute it.

Not a one-time bug. A pattern. Gemini Flash hallucinates instructions — not small details, entire workflows. "Only use the tools provided" gets ignored for "I'll just guess the answer." "Never write files" becomes a full file-write operation. Every correction I added made it worse because the model couldn't hold both the instruction and the constraint in its limited reasoning window.

I switched to DeepSeek. The difference was immediate: it follows instructions. It doesn't infer a better instruction. It does exactly what it's told. That's the minimum bar for an autonomous system, and Gemini Flash failed it repeatedly.

Then I tried Ollama — running models locally. The theory is sound: no rate limits, no data leaving your machine, full control. The practice: my machine isn't strong enough for the models that matter. Llama 3 70B doesn't run smoothly on consumer hardware. You need serious compute for serious local models.

Where I landed: A multi-model router. The orchestrator classifies every request into one of four buckets — classification/parsing (cheap), complex reasoning (expensive), code generation (specialized), trivial tasks (local). Each bucket routes to the right model. I ran the numbers: my Gemini Flash "savings" were an illusion. Every $1 I saved on inference cost me $4 in debugging time, regenerated outputs, and trust in my own system. Cheap models aren't cheap — they're expensive in a currency no billing dashboard shows.

What Actually Worked

Between all these failures, one architecture emerged: a layered system with built-in failure tolerance.

The agent doesn't do everything. It orchestrates everything. The email transport can swap providers without touching the agent logic. The model layer can add or remove models without reconfiguring the routing. The health layer monitors everything and feeds failures back into a self-improvement engine.

When the email provider changed its SMTP settings last week, the orchestrator detected the failure, rerouted through the backup channel, and logged the change — without me knowing until the morning report.

That's the real architecture: not an agent that never breaks. An agent that gets better at handling breakage each time.

You can't build that with a single model, a single provider, or a single attempt. You build it by failing five times and keeping the pieces that survived.

If you're building agent infrastructure and hitting the same walls — you're not alone. Every agent builder I talk to is making similar mistakes on different stacks. The transport layer fails first. Then the coordination layer. Then the model layer. The patterns are the same.
