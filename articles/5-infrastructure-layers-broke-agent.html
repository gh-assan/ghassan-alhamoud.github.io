The 5 Infrastructure Layers That Broke My Agent

AI Agents Infrastructure Architecture Post-Mortem AI OpenClaw

Everyone talks about building agents. No one talks about what breaks them.

I've been building an autonomous orchestrator — an agent that manages pipelines, reads emails, coordinates sub-agents, and improves itself. On paper, the architecture is elegant. In practice, 5 separate infrastructure layers failed over 3 months. Not "needed tuning." Spectacularly, account-deleting, model-hallucinating, IDE-fighting failures.

Here's the structural post-mortem. Each layer broke in a specific, predictable way — and each one demanded a counter-architecture that the industry is only starting to build.

Total: 6 weeks of work that a better upfront architecture could have saved.

Layer 1: Transport — Why Google Killed My Agent's Account

Email is the universal connectivity protocol for agents. Invoices, notifications, approvals — it all flows through SMTP. So I built on Gmail, the most mature email infrastructure on Earth.

Three weeks in: Google terminated my account. 50+ connected services, dead overnight. Why? Autonomous email access triggers every anti-abuse heuristic in their stack: unusual login patterns, high send frequency, automated reads. Google's security model assumes a human operator. An agent looks like a bot — because it is one.

Microsoft was worse. Hotmail's authentication flow demands periodic manual re-authentication via authenticator app. An agent can't tap "Approve" on a push notification. Three days in, my agent was locked out while I was asleep.

The fix: GMX + Thunderbird MCP. A boring German provider with no bot-detection hysteria, paired with a local MCP bridge. The agent never talks to SMTP. It talks to Thunderbird, which talks to the provider. A thin proxy layer — but that proxy is the difference between a terminated account and one running uninterrupted for days.

Architecture principle: Never let your agent touch external infrastructure directly. Insert a normalization layer that transforms agent behavior into something the provider tolerates.

Layer 2: Presentation — The Diagram Blind Spot

I wanted Mermaid diagrams generated automatically from architecture documents. The agent reads the architecture, produces a diagram, everyone wins.

The diagrams were terrible. Not syntax errors — the layout was wrong. Dependencies in the wrong direction. Boxes overlapping. Flows that made sense to a language model but not to a human reader.

Root cause: LLMs don't see visual output. They generate token sequences that statistically correspond to "diagram." There's no feedback loop — the model can't look at its output, go "that line crosses unnecessarily," and adjust. This is a fundamental limitation: text-only models can reason about spatial layouts in theory, but cannot critique them visually.

The fix: Two-stage pipeline. Stage 1 generates Mermaid syntax from source documents. Stage 2 renders it and feeds the image to a vision-capable model for critique. The critique feeds back into Stage 1. Repeat until quality thresholds pass. Three iterations minimum per diagram. Slower, more tokens — but it's the only way to close the visual feedback loop.

Architecture principle: Any output that a human would evaluate visually must be verified by a vision-capable model. Text-only evaluation is blind to its own blind spot.

Layer 3: Control — The Self-Deployment Dependency Trap

If the agent manages everything else, shouldn't it manage its own deployment? Remote provisioning — SSH in, install dependencies, configure services, start the daemon.

I spent 4 days on this. The sequence went: SSH → authenticate → discover environment → install deps → configure → test. Every step broke differently.

The agent couldn't install Docker because it didn't have root. It couldn't get root because it was running as a limited user. It couldn't change the user because it didn't have sudo. Each fix unblocked the next blocker — which was also broken.

On day 3, I SSHed into a machine that hadn't been patched in 18 months. The agent tried to install Node 22. The machine was running Ubuntu 18.04. The package manager was so outdated the install script threw an error for a dependency library that didn't exist when that OS version was released.

The fix: Separate deployment from operation. The agent handles operation — health checks, environment monitoring, restart on failure. A human-managed DevOps pipeline handles deployment (Ansible, Terraform, whatever you know). The agent can request changes by filing structured tickets. It does not execute them. Less "autonomous." Drastically less broken.

Architecture principle: The agent's control scope must exclude any operation that requires escalated privileges or environment-level changes. Define the boundary explicitly — and make it one-way (agent can observe, can't mutate).

Layer 4: Coordination — When the Agent Fights the IDE

This was the most hyped integration and the most frustrating failure. The agent controlling VSCode — opening files, running commands, managing Copilot.

What happened: the agent and VSCode started fighting each other. The agent opens a file. VSCode auto-formats it. The agent adds a comment. Copilot suggests a completion that overwrites it. The agent tries to refactor. The IDE auto-completes something else. Two reasoning systems on the same surface area produced a coordination cascade that neither could resolve.

Root cause: VSCode is designed for a single intelligent actor. Its API surface assumes one human operator. Adding a second actor creates shared-state conflicts that neither system is designed to handle. Copilot doesn't know the agent exists. The agent doesn't know Copilot is about to overwrite its changes.

The fix: The IDE is a tool, not a peer. The agent doesn't "control VSCode." It submits file patches, runs commands through a managed terminal session with mutex locking, and receives diffs through a structured output channel. One actor at a time. Serialized. Explicit. No surprises.

Architecture principle: Two reasoning agents operating on the same surface area requires a conflict-resolution protocol. If the tool doesn't provide one, the integration must be serialized to single-threaded access.

Layer 5: Model — The Wrong Model Breaks Everything

I started with Gemini Flash as the primary model. Cheap, fast, available.

On day 2: "Read this email and summarize it. Never generate code." Gemini generated a 200-line Python script and tried to execute it.

Not a one-time bug — a pattern. Every correction made it worse because the model couldn't hold both the instruction and the constraint in its limited reasoning window. "Only use tools" became "I'll just guess." "Never write files" became a full write operation. The model wasn't failing — it was succeeding at being a fast, cheap model with no depth.

I switched to DeepSeek. It follows instructions. It doesn't infer a better instruction. It does exactly what it's told. That's the minimum bar for an autonomous system.

The fix: Model selection isn't a stack choice — it's a routing problem.

Task ClassModelWhy
Classification & routingDeepSeek ChatFast, cheap, follows instructions
Complex reasoningDeepSeekBest architectural understanding
Code generationClaudeLowest hallucination rate for code
Trivial parsingLocal OllamaFree, private, good enough

I ran the numbers: my Gemini Flash "savings" were an illusion. Every $1 I saved on inference cost me $4 in debugging time, regenerated outputs, and trust in my own system. Cheap models aren't cheap — they're expensive in a currency no billing dashboard shows.

Architecture principle: One model per task, not one model per system. The routing layer classifies each request (a low-dimensional decision) and dispatches it. The routing cost is negligible — the savings from matching model capability to task complexity compound immediately.

The Architecture That Survived

Between these failures, one structure emerged:

  • A router that classifies every incoming task
  • Specialized sub-agents for each domain (email, code, monitoring, articles)
  • A memory pipeline that persists between sessions
  • A health layer that monitors all infrastructure
  • A self-improvement engine that turns every failure into a system fix

Every layer is replaceable. The orchestrator doesn't "own" any infrastructure — it communicates through well-defined interfaces. The email transport swaps providers without touching agent logic. The model layer adds or removes models without reconfiguring the orchestrator. The health layer monitors everything and feeds failures back into the improvement engine.

Last week, the email provider changed its SMTP settings. The orchestrator detected the failure, rerouted through the backup, and logged the change — without me knowing until the morning report. That's the pattern: not an agent that never breaks. An agent that gets better at handling breakage each time.

The agent isn't the software. The agent is the architecture.

I'm documenting this because every agent builder I talk to is hitting the same walls — just on different stacks. The patterns are the same. The transport layer fails first. Then the model layer. Then the coordination layer. If you're building agent infrastructure and any of this resonates, you're not doing it wrong. You're finding the edge of what works today.
