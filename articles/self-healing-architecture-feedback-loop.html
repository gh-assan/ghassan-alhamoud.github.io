Three Layers Is Not Enough - You Need a Feedback Loop That Closes

Self-Healing Architecture Feedback Loops

Every incident management pipeline I've seen is linear.

Detect → Triage → Fix → Postmortem → Archive. Each step feeds the next. Nothing feeds backward. The postmortem sits in a folder. Detection thresholds stay the same. The same failure, in slightly different wrapping, triggers the same cycle next sprint.

This isn't resilience. It's a paper trail.

The difference between a system that learns from failure and one that merely records it is a single architectural decision: does the output of your resolution pipeline update the input of your detection pipeline? If the answer is no, you have documentation, not healing.

Here is the architecture that changed how I think about this.

Pattern Detection (Not Aggregation, Feedback)

Individual failures are noise. Patterns are signal. The difference between the two is a threshold.

The pattern detector runs after every healing layer event. It reads the state file and evaluates three rules:

  • Same error type, same file, 3 hits → file a minor issue with full context (error, stack trace, path, count)
  • Same error type, same tool, 5 hits → file a major issue
  • Same error type across all tools, 10 hits → file a major issue and flag for automated postmortem

The thresholds are not arbitrary. They came from observing the system's failure distribution over two weeks of operation. The first 1-2 occurrences of any error type are almost always transient—latency blips, race conditions, cache staleness. By the third hit on the same file, the failure is deterministic. By the fifth, it is structural. By the tenth, the architecture has a class of failure that no amount of retries will fix.

The key architectural property here: the detector files issues programmatically into the same pipeline as human-filed tickets. There is no separate "self-healing" queue. There is no special handling path. The issue enters the standard workflow with a reported_by: "healing_layer_auto_detect" tag. This is critical because it prevents the self-healing system from becoming a black box that silently handles failures without accountability.

Root Cause Resolution (Not Postmortem, Heuristic Update)

When the pattern detector crosses a threshold, a full issue is created with five sections:

1. Signal data. Error type, file path, tool name, occurrence count, first and last timestamp, full session trace. This is the evidence package. Any engineer can look at this and reproduce the failure in under a minute.

2. Structured 5 Whys. Formal root cause analysis with a mandatory link between each level. Each "why" must directly cause the previous answer. If there is a logical gap, the analysis is rejected and must be re-run. This is enforced by a validation gate, not by convention.

3. Validation gate. Four checks, all automated:

  • Reproducibility. Can we trigger this exact failure on demand? If no, the analysis is speculative and discarded.
  • Archive match. Has this or a similar root cause been recorded before? If yes, the previous fix was insufficient and this issue is escalated—not for blame, but because the fix strategy needs to change.
  • Logical completeness. Every link in the 5 Whys chain has a corresponding action item in the fix plan. No orphan causes.
  • Counterfactual. If this fix had been applied before the first occurrence, would it have prevented all subsequent occurrences? If no, the root cause is still at the surface level.

4. Fix and verify. The fix is applied. The same failure scenario is replayed. If the failure no longer occurs, verification passes. If it does, the issue is reopened and the analysis chain is refined.

5. Heuristic update. The detection thresholds are adjusted. If stale edit failures hit 5 before triggering a major issue and we determined the correct threshold should have been 3, the heuristic is updated globally. This is the feedback loop—the output of resolution updates the input of detection. The next time a similar pattern starts forming, it is caught two occurrences earlier.

This is the difference between a postmortem and a heuristic update. A postmortem records what happened. A heuristic update changes what the system will catch next time.

The Edit Case: 37 Occurrences of the Same Failure

In one session, the healing layer caught 37 edit failures from stale oldText. The pattern was simple and deterministic: the agent read a file, the file's content changed in an unrelated operation, the agent attempted a targeted edit using the original text, the edit tool could not find the text, and the agent retried—with the same stale data.

Without the healing layer, these would have been logged as transient retries. With the healing layer, the pattern detector hit the 5-occurrence threshold at the 5th failure and filed issue ISU-004.

The 5 Whys chain:

  1. Why did the edit fail? — The oldText did not match the current file content.
  2. Why did oldText not match? — The file was modified between the read and the edit.
  3. Why was the staleness not detected? — No staleness check existed on the edit target.
  4. Why was there no staleness check? — The tool response handler had no channel to detect this pattern.
  5. Why did the architecture not provide this channel? — Tool calls were treated as atomic and independent operations. The response handler had no mechanism to detect inter-call state divergence.

Root cause: No feedback loop for the system's own execution health.

The fix had two parts. First, the healing layer became SOP for every tool call handler—not just edit operations. Second, the pattern detection heuristic was tightened: stale edit detection threshold reduced from 5 to 3 for all file operations.

Since that issue was closed, the stale edit failure pattern has not recurred. Zero occurrences. Not a single one.

A postmortem would have recorded this. A heuristic update eliminated it.

Why This Pattern Is Rare

Most systems stop at the monitoring layer. Dashboards, alerts, log aggregators—these are passive. They record what happened. They do not change what the system will detect.

The missing architectural component is the backward edge from resolution to detection. Most pipelines are directed acyclic graphs. The self-healing architecture is a graph with a cycle—the output of Layer 3 feeds back into the input of Layer 2 and Layer 1. This is not a best practice. It is a structural requirement for a system that improves over time.

The reason most teams do not build this is not technical. A three-layer intercept with a state file and a pattern detector is about 200 lines of code. The reason is architectural: it requires conceiving of tool responses as sensor data, not just results. Every tool response is also a measurement of system health. If you throw that measurement away, you cannot heal.

Where to Start

One sensor. One gate. One loop.

Instrument a single recurring failure pattern. Build a healing layer that catches it. File an issue when the threshold is crossed. Close the loop by updating detection based on what you learned.

Do this once. Observe the result. Then expand to the next pattern.

The architecture is cheap to start and expensive to ignore—because the first failure you did not catch will cost more than the entire system would have taken to build.
