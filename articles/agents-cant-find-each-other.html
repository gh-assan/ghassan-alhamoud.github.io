Agents Can't Find Each Other

The multi-agent discovery problem is the hard part nobody's solving.

I've been building multi-agent systems for two years. And the hardest problem isn't prompt engineering, tool use, or reasoning chains. It's discovery.

When Agent A needs help from Agent B, how does it find Agent B? How does it know what Agent B can do? How does it verify that Agent B is available, trustworthy, and competent for the specific task?

Right now, most systems solve this by hardcoding agent relationships. Agent A always calls Agent B. But that doesn't scale. In a real multi-agent system, agents should discover each other dynamically, the way microservices discover each other through service registries.

The Problem with Hardcoded Agent Topologies

  • Brittleness — If Agent B goes down, Agent A is stuck
  • No redundancy — What if Agent C could do the job better?
  • No load balancing — All traffic goes to Agent B, even when it's overwhelmed
  • No versioning — Agent B's capabilities evolve, but Agent A's references don't

Hardcoded agent graphs are the equivalent of hardcoded IP addresses. They work in a demo. They fail in production.

A Protocol-Based Approach

Here's what I'm building and recommending to clients: a Skill Registry Protocol.

Each agent publishes its capabilities to a registry:

Agent B registers:
  - skill: "text_summarization"
  - input: { text: string, max_length: number }
  - output: { summary: string }
  - quality: { accuracy: 0.94, latency_avg: "1.2s" }
  - status: "available" | "degraded" | "offline"

Agent A queries the registry:

Agent A asks: "Who can summarize text?"
  → Registry returns: [Agent B, Agent D, Agent E]
  → Sorted by: quality score ∧ availability ∧ latency

Agent A selects the best match and initiates a handshake with a contract:

Contract: { task_id, input_schema, output_schema, SLA, timeout }

This is the A2A (Agent-to-Agent) protocol that everyone's been talking about but nobody's actually implementing correctly. I've built this. It works.

The Hard Parts

Discovery is the easy part. The hard parts are:

  • Trust — How does Agent A trust Agent B's quality claims? (Reputation systems, verified execution logs)
  • Contracts — How do you enforce the terms of an agent handoff? (Formal verification, timeout-based fallbacks)
  • Discovery of new capabilities — How does the system adapt when a new agent comes online with better capabilities? (Registry events, blue-green deployment for agents)

These are the problems I solve for clients. If you're building multi-agent systems, you'll face them too.

The Takeaway

Don't hardcode your agent relationships. Build a discovery protocol from day one. Your agents will thank you — and so will your production on-call rotation.
