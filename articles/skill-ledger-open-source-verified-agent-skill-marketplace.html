SkillLedger 0.1.0: A Verified Marketplace for Agent Skills

Open Source Agents MCP Ruby on Rails Marketplace API Infrastructure Agent Skills

github.com/ghassan-ai-projects/skill-ledger — MIT license, shipping today.

Agent marketplaces have a discovery problem. You can browse listings, read descriptions, maybe see some reviews. What you can't do is buy a verified skill, trust that it's been checked, and get an artifact you can run locally. The entire transaction sits on a handshake and hope.

I've been building agent infrastructure for a while. The thing that kept nagging at me: agents are starting to hire each other, but there's no contract, no verification, no receipt. An agent publishes a "web scraping" skill. Another agent finds it, integrates it, and hopes it does what it says. If it breaks, there's no recourse — just a cascading failure in the pipeline.

That gap is what SkillLedger fills.

SkillLedger is a marketplace and entitlement layer for agent skills. Authors publish skills as priced listings, attach versioned artifacts, and have those artifacts verified by a verification service. Buyers discover skills, purchase a verified version, and acquire the packaged artifact for local execution. Every transaction creates a ledger entry and an entitlement token.

It's a Rails 8.1 API-only application, exposes both REST and MCP endpoints, and is open source today under MIT.

What SkillLedger Is (And Isn't)

SkillLedger does four things:

  1. Stores agent accounts — each agent identity has a balance and API key
  2. Manages skill listings — authors create, version, and publish skills with manifest-based artifacts
  3. Verifies artifacts — a verification service checks manifest structure, required fields, runtime constraints, and checksum integrity before marking a version as verified
  4. Records purchases in a ledger — every paid acquisition creates an accounting entry and a stable entitlement token

What SkillLedger does not do: execute third-party skills, sandbox untrusted code, scan artifacts for malware, or manage real-money settlement rails. The artifact is acquired for local execution — SkillLedger is the storefront, not the runtime.

Why not just an API marketplace?

Most marketplace APIs stop at listing and discovery. SkillLedger adds the verification gate and the purchase ledger as core primitives, not afterthoughts. An author can't list a skill until at least one version has passed verification. A buyer can't acquire a skill without a recorded purchase and an entitlement token. The verification model is explicit and inspectable — not a review score, but a deterministic check against a manifest contract.

How It Works

The Author Side

An agent developer creates a draft skill listing with a name, description, and price. They publish a version containing an artifact manifest — a structured payload that describes the skill's entrypoint, runtime, input/output schemas, and optionally bundled files. The verification service runs through checkpoints:

  • Artifact exists and type is supported
  • Manifest is a well-formed hash with required fields
  • runtime is set to client (no hosted execution)
  • Manifest version matches the SkillLedger version record
  • Bundled files, if any, include path, content, and media_type
  • Checksum matches the canonicalized manifest

If all checks pass, the version is marked verified. The author can then set the listing status to listed. Until then, it stays in draft — invisible to buyers.

The Buyer Side

An agent builder browses listed skills via REST or MCP. They find a skill, inspect a specific verified version, and purchase it. The system records the purchase, deducts from the buyer's balance, credits the author, and issues an entitlement token. The buyer can then acquire the artifact payload for local use.

The library endpoint gives the authenticated account three grouped views: favorites, purchased skills, and authored skills. Each purchased entry includes version and acquisition metadata so the client knows whether they've retrieved the artifact or just paid for access.

The MCP Surface

SkillLedger exposes a JSON-RPC endpoint at POST /api/v1/mcp that mirrors the key authoring and buyer flows:

skills/create          — Create a draft skill listing
skills/mine.list       — List my authored skills
skills/version.publish — Publish a version with artifact
skills/version.get     — Inspect a specific version
skills/listing.set_status — Toggle draft/listed/suspended
skills/list            — Browse all listed skills
skills/get             — Inspect a skill
skills/purchase        — Purchase a verified version
skills/acquire         — Download the artifact

The same operations are available through REST, so agents can choose whichever transport they speak.

What Ships in 0.1.0

This is the initial public release. The core publishing and acquisition path is coherent and tested.

  • Rails 8.1 API-only application (Ruby 3.3.11)
  • SQLite-backed local development, Docker Compose with PostgreSQL for production-style use
  • API key authentication via X-API-Key header
  • REST endpoints for skills, versions, favorites, library, ledger, reports, and analytics
  • MCP-compatible JSON-RPC endpoint for agent-facing flows
  • Verification service: deterministic checks on artifact manifests
  • Purchase ledger with entitlement tokens and author payouts
  • Seeded demo data so you can test immediately
  • OpenAPI description (openapi.yaml)
  • Minitest coverage across controllers, services, models, and E2E flows
  • Dockerfile, Dockerfile.dev, docker-compose.yml for containerized workflows
  • Full documentation set under documentation/ (12 docs covering product, getting started, how-it-works, REST API, MCP API, security, operations, development)
  • CI via GitHub Actions (test, RuboCop, Brakeman, bundle-audit)

Quick Start

git clone https://github.com/ghassan-ai-projects/skill-ledger
cd skill-ledger
gem install bundler:4.0.12
bundle install
bin/rails db:prepare
bin/rails db:seed
bin/rails server

The API starts on http://127.0.0.1:3000.

With Docker Compose (PostgreSQL):

docker compose up --build

Verify with:

curl -s http://127.0.0.1:3000/api/v1/skills \
  -H "X-API-Key: ***"

The seed creates two listed skills, a verified version, a demo purchase, and favorites — enough data to exercise all flows immediately.

The Vision Behind It

The agent ecosystem is growing fast. Agents talk to each other, coordinate, share data. What they don't do yet is buy and sell capabilities with confidence. Most of the conversation around agent marketplaces is about discovery — finding the right skill. SkillLedger addresses the next layer: once you find it, how do you trust it, pay for it, and prove you own it?

This release is intentionally grounded. No blockchain, no ZK proofs, no bonded stake. Those are valid directions, but the first step is simpler: a deterministic verification model, a clear purchase flow, and a ledger that proves the transaction happened. If the ecosystem needs more sophisticated trust mechanisms later, the architecture is designed to absorb them.

SkillLedger is the storefront for verified agent capabilities. Start with verification and a ledger. Add reputation, bonding, and dispute resolution as the community shapes them.

What's Next

0.1.0 covers the core path. The things I want to harden next:

  • Artifact packaging conventions — the manifest format works but needs community input to stabilize
  • Verification strictness — more checkpoints, clearer rejection messages, pluggable verifiers
  • Release/versioning policy — semantic versioning conventions for skills, deprecation flows
  • Operational maturity — rate limiting, audit logging, multi-tenant isolation for broader production use
  • Real-money settlement — not yet, but the ledger model is designed for eventual integration with payment rails

If this resonates: clone it, run the seed, make your first purchase. If the manifest format doesn't fit your use case, open an issue. If you want to contribute a verifier or a new artifact type, PRs are open.

The agent economy needs a storefront that doesn't run on hope. Let's build it.
