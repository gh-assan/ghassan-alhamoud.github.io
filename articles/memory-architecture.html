Every Token Is Rent

Context windows are a trap. Every token you keep past its usefulness is rent you're paying for no reason.

I see teams building LLM systems with increasingly large context windows as if they're a solution to memory. "We'll just throw everything into the context!" they say. "It's cheap!"

It's not cheap. Every token costs you in three ways: latency, quality, and money. And the bigger your context gets, the worse all three become.

The Three Costs of Context

Let's break down why "just use more context" is an architectural anti-pattern:

1. Quadratic Attention

Transformer attention scales O(n²) with context length. Doubling your context doesn't double your cost — it quadruples it. The difference between 32K and 128K tokens isn't 4x more expensive. It's 16x more expensive in compute.

2. The Lost Needle Problem

Research consistently shows that LLMs perform worse at finding relevant information as context grows. The "needle in a haystack" problem is real. More context doesn't mean better recall — it often means worse.

3. Diminishing Returns

In every LLM system I've audited, 80% of the context window is irrelevant noise. Historical conversation turns that will never be referenced again. Documents that were included "just in case." System prompts that have grown to novella length.

The Right Architecture: Tiered Memory

Instead of a single bloated context, use a tiered memory architecture:

Level 0 (Working Memory): Current conversation, ~4K tokens
  ↕
Level 1 (Episodic Memory): Recent sessions, summarized
  ↕
Level 2 (Semantic Memory): Knowledge base, indexed
  ↕
Level 3 (Long-term Storage): Raw logs, archived

Each level has a different retrieval strategy. Level 0 is always present. Level 1 is retrieved based on conversational triggers. Level 2 requires explicit search. Level 3 is offline.

The key insight: most interactions only need Level 0. The system doesn't carry the past — it fetches it on demand.

Practical Implementation

Here's how I implement this in production systems:

  • Summarization agent — After every session, the system generates a structured summary (not a raw transcript) and stores it in a vector database
  • Trigger-based retrieval — When a user references a past conversation, the system retrieves only the relevant summary, not the entire history
  • Context pruning — Every 10 turns, the system evaluates whether each token in the current context is still earning its keep. Noise gets evicted.
  • Budget-aware scheduling — Each interaction has a token budget. If you blow it on context, you have less for generation.

Memory isn't free. Every token is rent. Build systems that pay rent only for what they actually use.
